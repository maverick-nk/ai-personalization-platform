# Retrospective — inference-api

**Date:** 2026-05-14  
**Service:** inference-api

---

## What went well

Handling of all external calls — MLflow model loading, Redis feature fetch, and the privacy service — was done correctly with async patterns. None of them block the request path: `asyncio.to_thread()` for blocking MLflow I/O, async Redis and httpx clients for network calls, and `asyncio.Lock` to coordinate hot-swap without stalling in-flight requests.

## Would design differently

Understand the client surface before committing to a protocol. If a service is strictly internal (service-to-service), Go + gRPC is a strong default — binary framing, generated contracts, low latency. When a service faces both end users and internal callers, REST reduces friction at the cost of some efficiency. The key is that this decision should be driven by who is calling the service and their tooling constraints — not protocol preference — and latency requirements must be weighed against the added complexity of certain stacks (e.g. CGO for LightGBM in Go).

## Most surprising

Nothing was surprising — the design was well-specified before implementation started and the asyncio model behaved as expected throughout.

## Open questions

How would model hot-swap be handled under live traffic in production? The current implementation uses `asyncio.Lock` to make the version-check-and-swap atomic within a single process. In a multi-replica deployment (Kubernetes pods), each replica independently polls MLflow and swaps at slightly different times — a request load-balanced during the swap window could hit replicas running different model versions simultaneously.
