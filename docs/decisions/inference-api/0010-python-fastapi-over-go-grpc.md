# 0010. Python + FastAPI REST over Go + gRPC for Inference API

**Date:** 2026-05-13  
**Status:** Accepted  
**Service:** inference-api  
**Decided by:** user

---

## Context

The original design specified Go + gRPC for the inference-api. Two constraints made this impractical. First, LightGBM has no first-class Go binding — loading a native `.txt` artifact from MLflow requires either CGO (complicating the build and Docker image) or exporting to ONNX (an extra pipeline step that creates a new artifact type diverging from what training produces). Second, the test harness uses pytest + httpx, which speaks HTTP natively, not gRPC framing. A REST-to-gRPC gateway would have been required to bridge the two, making gRPC a layer of overhead that doesn't reach the actual caller.

---

## Decision

The inference-api is implemented in Python + FastAPI serving REST over HTTP. The <50ms latency target is still achievable using async I/O (httpx for the privacy check, redis-py async for feature fetch) and asyncio.to_thread() to offload blocking MLflow I/O off the event loop.

---

## Alternatives Considered

| Option | Why rejected |
|---|---|
| Go + gRPC (original plan) | No first-class LightGBM Go binding; CGO or ONNX export required to load model artifacts |
| Go + gRPC + REST gateway | Adds a second protocol surface; gateway behaviour under edge cases can diverge from the gRPC endpoint; test harness still only exercises the gateway layer |
| ONNX export + Go inference | Adds an artifact conversion step in the training pipeline; diverges artifact type from what MLflow registers; conversion correctness is a new failure mode |
| Python + gRPC | Eliminates the Go/LightGBM binding problem but still requires a gateway for httpx-based tests, and protobuf savings (~10–20% payload) are negligible next to Redis and model inference time |

---

## Consequences

**Gets easier:**
- LightGBM artifacts load directly via `mlflow.lightgbm.load_model()` — no conversion step, no CGO
- pytest test harness (httpx) speaks the same protocol as the service — no gateway layer
- OpenAPI spec is auto-generated by FastAPI; no `.proto` files to maintain
- Single service process with no sidecar; Docker image is a standard Python image

**Gets harder / trade-offs accepted:**
- Binary framing efficiency of Protobuf is not realized — JSON payloads on the recommendation response are larger, though this is negligible at Top-N scale
- No generated type-safe client stubs — callers rely on the OpenAPI spec instead of a compiled proto contract
- Python's throughput ceiling under CPU-bound load is lower than Go's; meaningful only if model inference becomes CPU-bound at high RPS

**Constraints this introduces:**
- Do not migrate to gRPC without first solving LightGBM artifact loading — the binding problem is the root constraint, not protocol preference
- Any future language change must keep the artifact loading strategy in sync with what the model-training pipeline registers in MLflow

## System Design Concepts

| Concept | DDIA Reference | How it applies here |
|---|---|---|
| RPC vs REST trade-offs | Ch 4 — Encoding and Evolution | Binary encoding (Protobuf) reduces payload size but the efficiency gain is dominated by other latency contributors (Redis, model inference); REST was chosen for test harness compatibility, not payload optimisation |
| Protocol encoding and schema evolution | Ch 4 — Encoding and Evolution | JSON is more verbose than Protobuf but schema-on-read compatible without generated stubs; OpenAPI provides the contract that proto files would otherwise enforce |
| Blocking I/O in async systems | Ch 11 — Stream Processing | MLflow artifact downloads are synchronous; asyncio.to_thread() offloads them to a thread pool to keep the event loop free for request handling — the same principle that makes Kafka consumers avoid blocking the main loop |
