# Retrospective — feature-pipeline

**Date:** 2026-04-23
**Service:** feature-pipeline

---

## What went well

Not recorded.

## Would design differently

With more services and consumers, Avro + schema registry would be a better choice for communicating schema between services. It enforces contracts at the producer/consumer boundary and removes the need for manual coordination when field names or types change.

## Most surprising

The lock drain pattern in `ParquetSink` — that a lock can be used to safely drain a buffer before closing, and without it events in-flight at shutdown would simply be dropped. The copy-then-release pattern (drain under lock, write outside lock) was a non-obvious but important correctness invariant.

## Open questions

Python packaging with `apache-flink` and `uv` had friction — `uv sync` removes manually installed packages, and JVM-dependent packages need special handling (`--no-build-isolation`). A Makefile was created as a local workaround, but there is no clean solution yet for making this reproducible in CI or Docker without manual steps.

---

## Follow-up actions

- **Avro + schema registry** is on the roadmap. When implemented, run `/adr` to record the decision — the trade-offs (schema evolution, registry as a dependency, cross-language contracts) are worth capturing formally. This is already partially covered in ADR 0004's alternatives section.

- **Python packaging friction** (`apache-flink` + `uv`): worth investigating a cleaner Docker/CI path before the productionization phase. Consider pinning a pre-built wheel or using a multi-stage Docker build that installs JVM packages separately.

- **Lock drain pattern** maps to fault tolerance in stream processing (DDIA Ch 11 — exactly-once vs at-least-once, checkpointing, shutdown semantics). Run `/concept-quiz` to test this if it still feels fuzzy.
