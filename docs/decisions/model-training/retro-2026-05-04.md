# Retrospective — model-training

**Date:** 2026-05-04
**Service:** model-training

---

## What went well

The model-agnostic `BaseTrainer` ABC design worked cleanly — the feedback to decouple algorithm concerns from orchestration was right, and the factory pattern means adding XGBoost later is a one-file change. The Parquet reader, feature engineering, and MLflow registration all worked first-try once the infrastructure issues were resolved.

## Would design differently

The MLflow experiment creation. Using `mlflow.set_experiment()` by name is fragile — if the experiment was created under a different artifact root (as happened here when the docker-compose config changed mid-session), the name is reused but the artifact location is wrong, and soft-deleting it makes the name unusable without a DB-level cleanup. The fix — `MlflowClient.create_experiment(name, artifact_location="mlflow-artifacts:/")` with an active-state check — is now in the code and should be the pattern for any future service that touches MLflow.

## Most surprising

**MLflow client/server version coupling.** MLflow 3.x client added a `/logged-models` API endpoint that doesn't exist in the 2.x server. The client resolved successfully but silently failed at artifact log time with a 404. Required pinning `mlflow>=2.14,<3.0` in `pyproject.toml` and reconfiguring the server with `--serve-artifacts --artifacts-destination /mlflow/artifacts --default-artifact-root mlflow-artifacts:/` so host-side clients can upload artifacts over HTTP rather than directly to the Docker volume. **This will affect inference-api (Step 5)** — it must use the same `<3.0` pin and set `MLFLOW_TRACKING_URI=http://localhost:5001` to download model artifacts.

## Open questions

- **AUC = 1.0 on synthetic data is expected** (label is derived from `avg_watch_duration`, which is also a training feature), but it means the model has learned a trivial identity. Real data from the feature pipeline will produce more meaningful metrics — worth re-running training after Step 6 (test harness) generates actual watch events.
- **Genre coverage at inference time:** genres in `feature_schema.json` are fixed at training time. If a user's `session_genre_vector` at inference contains a genre not seen during training, that column will be absent. The inference-api needs a strategy — either zero-fill unknown genres (safe) or reject requests with out-of-schema genres (strict). Not yet decided.
- **Model signature not logged:** MLflow warned that no input signature or example was provided. This would let MLflow validate feature shapes at load time. Low priority for now but useful for catching training/serving skew before inference.
