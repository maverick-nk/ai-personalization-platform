# Retrospective — shared infra bootstrap

**Date:** 2026-04-12  
**Service:** shared (infra)  
**Step:** Step 0 — docker-compose baseline

---

## Most surprising

Two concepts clicked during this step:

**Redis persistence:** The reason we disable both RDB and AOF is not laziness — it's a deliberate architectural choice rooted in the derived data principle. Features are a materialised view of the Kafka event log, not a system of record. If Redis loses its data, the feature pipeline can replay from Kafka and reconstruct the full feature store. Persistence would add I/O overhead for data that is, by design, always recomputable.

**Liveness vs readiness:** Liveness checks are well-known ("is the process alive?"), but readiness is the more operationally important probe. A service can be alive — passing liveness, not getting restarted — while being unable to serve correct traffic: model not yet loaded, upstream dependency not ready, internal queue backed up. Readiness gates traffic independently of liveness, which is critical for the inference API during model hot-swap.

## Follow-up actions

- The derived data / Kafka replay pattern will re-surface in Step 2 (feature pipeline). The idempotency problem under at-least-once delivery is the key design challenge there — flagged in the concept quiz.
- Liveness vs readiness will need to be explicitly wired in Step 5 (inference API) during model hot-swap implementation. The readiness probe should fail until the first model version is loaded from MLflow.
