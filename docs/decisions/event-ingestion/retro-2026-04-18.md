# Retrospective — event-ingestion

**Date:** 2026-04-18
**Service:** event-ingestion

---

## What went well

The API service was kept simple and user privacy was handled correctly from the start — raw `user_id` is never logged or persisted, only the HMAC-SHA256 pseudonym reaches Kafka. The deliberate constraint of no debug logging of PII made the privacy boundary clean and easy to audit.

## Would design differently

Nothing — this was a planned implementation following the architecture design. No surprises that changed the approach.

## Most surprising

The fire-and-forget Kafka publish model (202 Accepted, no delivery confirmation to the caller) has a direct impact on latency numbers. The insight here is that for watch and session events, occasional message loss is acceptable: a missed event won't drastically alter a user's recommendations, and the model tolerates sparse data gracefully. This makes fire-and-forget the right trade-off, not just a shortcut.

## Open questions

- **Containerization:** The service needs a Dockerfile and to be wired into `docker-compose.yml` as a running service (currently only infra is containerized).
- **Avro schema definitions:** Event schemas are currently Pydantic models local to this service. Moving them to Avro definitions in a shared schema registry would let downstream consumers (Flink, feature pipeline) know the schema at consume time — important for contract enforcement across pipelines.
